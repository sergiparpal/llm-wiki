llm-wiki Plugin — Fix Plan

 Context

 The llm_wiki plugin at /home/sergi/llm-wiki/ implements Karpathy's LLM Wiki pattern as a Hermes Agent memory provider. It was authored against documentation
  only and has never been run against the actual Hermes runtime (cloned at /tmp/hermes-agent/).

 Review against the canonical MemoryProvider ABC (/tmp/hermes-agent/agent/memory_provider.py), MemoryManager (/tmp/hermes-agent/agent/memory_manager.py), and
  the bundled honcho/mem0 providers found 36 issues. Several BLOCKERs prevent the plugin from loading or operating at all; gate-side path-prefix bugs make
 the quality gate effectively non-functional even on synthetic input.

 Goal: bring the plugin to a working, correct state, then clean up. Apply phases in order — earlier phases unblock later ones.

 Reference for "what right looks like":
 - /tmp/hermes-agent/plugins/memory/honcho/__init__.py — flat tool schemas, register(ctx) pattern, tool_error(), on_pre_compress, sync_turn threading.
 - /tmp/hermes-agent/plugins/memory/__init__.py:41-47 — plugin discovery scans $HERMES_HOME/plugins/<name>/ (no memory/ subdir).
 - /tmp/hermes-agent/run_agent.py:1699-1713 — Hermes wraps flat schemas into OpenAI shape itself.
 - /tmp/hermes-agent/run_agent.py:8699-8700 — handle_tool_call(name, args) is invoked with two args today, but ABC contract requires **kwargs for forward
 compatibility.

 ---
 Phase 1 — BLOCKERs that prevent loading or basic operation

 After this phase the plugin should load, dispatch tools, run prefetch and on_pre_compress without crashing, and survive the first ingest.

 1.1 __init__.py — wrong tool schema shape

 Where: __init__.py:167-168 and helper _fn at 271-283. Schemas are emitted in OpenAI-wrapped form ({"type": "function", "function": {...}}). Hermes wraps
 flat schemas itself and will double-wrap, resulting in a malformed tool list at API call time.
 Fix: Replace _fn with a flat dict builder that emits {"name", "description", "parameters": {"type": "object", "properties": ..., "required": ...}}. Pattern:
  /tmp/hermes-agent/plugins/memory/honcho/__init__.py (PROFILE_SCHEMA, SEARCH_SCHEMA, CONCLUDE_SCHEMA).

 1.2 __init__.py — wrong lifecycle hook name and signature

 Where: __init__.py:221-232 defines on_compress(messages, compression_count). The ABC method is on_pre_compress(messages) -> str (no count, returns string).
 This override never fires.
 Fix: Rename to on_pre_compress(self, messages: list[dict[str, Any]]) -> str. Read compression_count from a counter file kept by the plugin (or pass 0 to
 flush_compression_to_raw and use a timestamp slug instead — see 1.2a).

 1.2a operations.py — flush_compression_to_raw signature must change

 Where: operations.py:456 requires compression_count. With the renamed hook there is no count.
 Fix: Either drop the parameter and slug with datetime.now(...).strftime("%Y%m%dT%H%M%S"), or persist a counter at wiki/.compression_count that the hook
 bumps. Prefer the timestamp option — simpler, no extra state.

 1.3 __init__.py — prefetch signature missing session_id

 Where: __init__.py:201 defines prefetch(self, query: str = ""). The ABC is prefetch(self, query: str, *, session_id: str = "") -> str. Hermes calls
 prefetch(query=..., session_id=...) and this override blows up with TypeError: unexpected keyword argument.
 Fix: Change to def prefetch(self, query: str, *, session_id: str = "") -> str:. Ignore session_id inside (single-tenant plugin).

 1.4 __init__.py — handle_tool_call missing **kwargs

 Where: __init__.py:170. ABC requires **kwargs. Today's caller (run_agent.py:8699-8700) passes positionally, but the contract is the ABC, and any caller that
  adds a kwarg breaks this provider in isolation.
 Fix: def handle_tool_call(self, name: str, args: dict[str, Any], **kwargs: Any) -> str:.

 1.5 __init__.py — register() doesn't follow plugin contract

 Where: __init__.py:384-386 returns the class. Canonical contract (per honcho/mem0) is def register(ctx) -> None:
 ctx.register_memory_provider(WikiMemoryProvider()). Today's loader has a class-walking fallback that masks the bug, but plugin.yaml's entry_point:
 __init__:WikiMemoryProvider is also non-canonical.
 Fix: Replace with def register(ctx): ctx.register_memory_provider(WikiMemoryProvider()). Drop entry_point: from plugin.yaml — providers are discovered via
 register(ctx).

 1.6 README.md — wrong install path

 Where: README.md:36-39 says ~/.hermes/plugins/memory/llm_wiki/. Loader scans $HERMES_HOME/plugins/<name>/ directly.
 Fix: Change to ~/.hermes/plugins/llm_wiki/ everywhere in the README. Verify with /tmp/hermes-agent/plugins/memory/__init__.py:41-47.

 1.7 gate.py + operations.py — gate path prefix mismatches (single root cause)

 The gate produces diff keyed by paths relative to the wiki root (e.g. wiki/entities/foo.md), but the deterministic checks treat the keys as if they were
 relative to wiki/. This makes every gate check silently miss its targets.

 Fix all six call sites in one pass:

 1.7a gate.py:148-151 — check_links existing-set construction

 Issue: The set is built from wiki_root.rglob("*.md") and keyed by relative_to(wiki_root).with_suffix(""). If wiki_root is the git root (parent of wiki/),
 keys look like wiki/entities/foo, raw/articles/..., and matches work for diff-resolved wikilinks. If wiki_root is the wiki/ subdir, keys look like
 entities/foo. The plugin's Store.root IS the git root (containing wiki/, raw/, SCHEMA.md), and gate.run is called with wiki_root=store.root
 (operations.py:397-403). So the existing-set keys are correct for that base, but wikilinks in pages are written as [[entities/foo]] (no wiki/ prefix;
 SCHEMA.md §3 uses bare slugs).
 Fix: Strip the wiki/ prefix from existing keys when adding (skip raw/ entries entirely — wikilinks shouldn't target raw). Confirm by writing one test ingest
  and observing the gate logs.

 1.8 operations.py:489-503 and operations.py:507-542 — same path-shape bugs in lint

 Where:
 - _quick_frontmatter_check line 492: any(s in rel for s in ("/index.md", ...)) uses substring, OK in principle but rel here is the full pathlib.Path
 stringified (returned by target of wiki_write). The check works by accident because /index.md will appear in wiki/index.md paths, but it is fragile and
 over-matches wiki/entities/index.md if a user makes one.
 - _lint_inventory line 510: builds existing with wiki/ prefix (since relative_to(store.root) includes the wiki/ segment). But wikilinks from pages drop the
 prefix — same bug as 1.7a.
 - _lint_inventory line 519: rel.endswith(("index.md", "log.md", "overview.md")) — over-matches entities/log.md if it ever exists.

 Fix: Centralize a _is_special(rel: str) -> bool helper that returns rel in {"wiki/index.md", "wiki/log.md", "wiki/overview.md"}. Strip wiki/ from existing
 keys for link resolution (mirror 1.7a), and use the helper in both _quick_frontmatter_check and _lint_inventory.

 1.9 operations.py:405 — ingest counter never increments correctly

 Where: slug.startswith(("ingest-", "")) — the empty-string prefix makes the predicate always True. Looks like leftover from a defensive pattern that was
 never finalized.
 Fix: Drop the prefix check. Bump counter unconditionally on non-REJECT verdicts: n = store.bump_ingest_count() if verdict.decision != "REJECT" else None.

 1.10 __init__.py:80 — lint_every_n_ingests=0 disables silently broken

 Where: int(config.get("lint_every_n_ingests") or 50). The or collapses 0 to 50, exactly inverting the documented "0 disables" behavior.
 Fix: raw = config.get("lint_every_n_ingests"); self._lint_every_n = int(raw) if raw not in (None, "") else 50. Then in line 183 the existing if
 self._lint_every_n: correctly skips when 0.

 Phase 2 — MAJORs (correctness, data safety)

 2.1 operations.py:132-139 — Store.resolve substring escape check

 Where: if not str(target).startswith(str(anchor)) — /wiki-data starts with /wiki, so a path resolving to a sibling directory passes. Classic
 prefix-collision.
 Fix: Use target.is_relative_to(anchor) (Python 3.9+). Or compare via os.path.commonpath([str(target), str(anchor)]) == str(anchor).

 2.2 operations.py:233-250 — wiki_capture commits with git add -A

 Where: wiki_capture calls store.commit(...) which runs git add -A. If the agent has uncommitted wiki/ edits in flight (mid-ingest), this sweeps them into a
 "capture" commit, breaking the gate workflow.
 Fix: In Store.commit, accept paths: list[str] | None = None and stage only those paths when given. From wiki_capture, pass the single new file. The gate's
 full-tree commit still uses add -A.

 2.3 gate.py:440-450 — _apply_edits is silently lossy

 Where: Returns applied > 0, which is True even if 9 of 10 fixes had not target.exists() and were skipped. The gate then commits with "(gate-edited)"
 claiming success.
 Fix: Return applied == expected where expected = sum(1 for f in findings if f.severity == "fix" and f.suggested_edit). Log skipped paths. On partial
 application, treat as REJECT.

 2.4 gate.py:78-92 — staged_diff mishandles renames

 Where: git status --porcelain=v1 -z emits renames as R<status> <new>\0<old>\0, so a rename consumes two NUL fields but the loop treats each as a separate
 entry. Status R paths get parsed as bare paths and before is computed against HEAD:<garbled>.
 Fix: Iterate with an index, peek next entry when status is R or C, and consume two fields. Reference: git status --porcelain=v1 -z documentation.

 2.5 No concurrency lock on the wiki tree

 Where: Two concurrent calls to wiki_commit (e.g., overlapping sessions sharing the same wiki) will race on git status / git commit and produce torn or
 interleaved commits.
 Fix: Add a file-based lock at store.root / ".gate.lock" using fcntl.flock (POSIX) or a simple lockfile-with-pid pattern. Acquire in wiki_commit; release on
 exit (try/finally). Single-process flag in Store is sufficient if we don't expect cross-process sharing — confirm with the user.

 2.6 __init__.py:99-127 — validator-error path makes the gate uncrossable

 Where: When agent.llm_client.call_model import succeeds but the call raises, _call returns a JSON block-severity finding. gate.resolve_verdict then yields
 REJECT. So any validator error rejects the ingest. Compare against a network blip: the user loses work because the validator API was flaky.
 Fix: Demote validator failures to a fix severity with no suggested_edit, and in _apply_edits skip findings without an edit (already does this). Result:
 validator outage → APPROVE_WITH_EDITS with zero edits applicable → REJECT only if there are no other approvable changes. Or simpler: log the failure, return
  empty findings, let deterministic checks decide. Choose based on user preference (CLARIFY).

 2.7 gate.py:298-303 — same as 2.6 for non-JSON validator output

 Same fix as 2.6 applies. Currently any non-JSON validator response blocks the commit.

 2.8 __init__.py:201-205 — prefetch swallows all exceptions

 Where: except Exception: return "" hides real bugs (e.g., Store not initialized, FS permission errors).
 Fix: Catch only expected errors (OSError, subprocess.CalledProcessError); log to logger.debug for diagnostics. Let programming errors propagate during
 development.

 2.9 agent.llm_client does not exist

 Where: __init__.py:111 imports from agent.llm_client import call_model. There is no such module in /tmp/hermes-agent/agent/. The gate will always fall
 through to the deterministic-only stub.
 Fix: Use Hermes's actual model-call entry point. Search hermes-agent for the canonical helper (likely something in agent/api_client.py or run_agent.py). If
 none is exported as a stable API, add a config field validator_command (shell command pipe) or document that the user must pass llm_call= themselves via a
 config hook. CLARIFY with the user — there is no clean public path here.

 Phase 3 — MINORs

 3.1 __init__.py:208-219 — dead enrich_turn and on_turn_complete

 Not in the ABC. Delete both methods.

 3.2 plugin.yaml — non-canonical key shape

 Compare to /tmp/hermes-agent/plugins/memory/honcho/plugin.yaml:
 name: honcho
 version: 1.0.0
 description: "..."
 pip_dependencies: [honcho-ai]
 hooks: [on_session_end]
 Drop type: memory_provider, entry_point:, dependencies: (use pip_dependencies:). Drop the in-yaml config: block — config schema lives in
 get_config_schema().

 3.3 operations.py:485-486 — replace local _err with tool_error

 Use from tools.registry import tool_error (mem0's pattern). Single, consistent error JSON shape across providers.

 3.4 operations.py:545-561 — _tail_log uses naive timestamps

 datetime.fromisoformat(m.group(1)) on a YYYY-MM-DD string is naive; comparing with datetime.now(timezone.utc).timestamp() mixes naive/aware. Fix: parse with
  .replace(tzinfo=timezone.utc) after fromisoformat, then .timestamp().

 3.5 gate.py:109 — FRONTMATTER regex doesn't accept CRLF

 \A---\n(.*?)\n---\n requires LF. Windows-authored pages will fail. Fix: \A---\r?\n(.*?)\r?\n---\r?\n.

 3.6 Comments that explain WHAT not WHY

 Several modules have docstrings/comments narrating the obvious. Drop or convert to one-liners stating why (constraint, invariant, prior bug). Examples:
 __init__.py:215-219, operations.py:152-154, gate.py:104-106. Apply only where a reader benefits; do not chase every comment.

 3.7 gate.py:453-455 _stub_llm and gate.py:458+ main() — dead in-plugin scaffolding

 The plugin always uses _build_validator_call, never _stub_llm; main() is unreachable when loaded as a module. Delete both. If a CLI is wanted, gate it on if
  __name__ == "__main__": only.

 3.8 __init__.py — missing system_prompt_block override

 A short pointer (e.g. "# LLM Wiki\nUse wiki_read/wiki_list to navigate; wiki_ingest → wiki_commit for new sources.") helps the agent find the tools. Mirrors
  mem0/honcho.

 3.9 operations.py:507-542 — _lint_inventory orphan rule too lax

 p.endswith("hermes-agent") matches any *hermes-agent path including entities/old-hermes-agent. Fix to p == "wiki/entities/hermes-agent".

 3.10 gate.py:344-351 — revert runs git clean -fd

 -fd removes untracked dirs. If the user dropped a local note in the wiki root before the gate runs, it's gone. Constrain to git checkout -- . plus git clean
  -fd -- wiki/ so capture-time files stay.

 Phase 4 — NITs

 4.1 Hardcoded literals — extract a few to module-level constants

 {"index.md", "log.md", "overview.md"} appears in three places (gate.py:120, operations.py:492, operations.py:519). Extract SPECIAL_PAGES. Same for
 REQUIRED_KEYS, VALID_TYPES.

 4.2 gate.py:Verdict — add to_dict() for clean serialization

 Currently the call site dataclass-asdicts it implicitly. Add an explicit method to pin the JSON shape.

 4.3 _seed_* functions in operations.py — single-source the seed templates

 Currently _seed_index, _seed_log, etc. are defined as standalone functions. Move to a seeds/ package directory or a single _SEEDS: dict[str, str] mapping.

 4.4 flush_compression_to_raw — currently captures user-only turns

 Rationale ("user side is the load-bearing material") is correct but unstated. Add a one-line comment, or accept a roles= parameter for future flexibility.
 Comment-only is fine.

 4.5 Type hints — consistency pass

 Mix of dict[str, Any] and unparameterized dict. Settle on PEP 585 (dict[str, Any]) since from __future__ import annotations is already imported in
 __init__.py.

 4.6 subprocess.run(...) ergonomics

 Several _git callsites duplicate the ["git", "-C", str(self.root), ...] prefix. The _git helper exists; use it consistently (some places in operations.py
 and gate.py shell out directly).

 4.7 Logging

 There is no module logger. Add logger = logging.getLogger(__name__) to each of __init__.py, operations.py, gate.py. Use logger.debug for swallowed
 exceptions and logger.info for gate verdicts.

 Open clarifications (block Phase 2 work)

 1. Validator client (2.9): There is no agent.llm_client.call_model in Hermes. Three options — (a) reuse Hermes's internal API client by importing a private
 path (brittle), (b) require the user to provide a validator_command shell command, (c) skip LLM review and ship deterministic-only. Which?
 2. Validator-error policy (2.6): When the validator API errors out, REJECT the ingest (current behavior — safe but loses work) or APPROVE on
 deterministic-only checks (permissive but doesn't lose work)? Recommendation: log + degrade to deterministic-only, since deterministic checks already cover
 the most common failure modes.
 3. Concurrency model (2.5): Is the plugin expected to handle multiple concurrent Hermes sessions sharing the same wiki dir? If single-tenant (likely, given
 this is a personal-knowledge tool), a per-process in-memory threading.Lock is enough; cross-process needs flock.

 Verification

 End-to-end smoke test after each phase:

 1. Install (after 1.6): cp -r /home/sergi/llm-wiki ~/.hermes/plugins/llm_wiki and run hermes config set memory.provider llm_wiki.
 2. Load (after 1.1–1.5): Start hermes; check stderr for plugin discovery success. hermes memory status should list llm_wiki as active.
 3. Tool dispatch (after 1.1, 1.4): From the agent, call wiki_capture with a small test source. Check that ~/.hermes/llm_wiki_data/raw/notes/<date>-test.md
 exists and has a single git commit.
 4. Ingest + gate (after 1.7, 1.8): Call wiki_ingest on the captured source; have the agent write a couple of pages; call wiki_commit. Watch the gate
 findings — broken-link, citation, frontmatter checks should now fire on actual mistakes (verify by injecting one). Without 1.7, every page passes the
 deterministic checks and you'll see "APPROVE" on garbage.
 5. Compression (after 1.2): Force a compression event (long session or manual /compact). Confirm a new raw/sessions/<timestamp>-...md lands and is
 committed. Without 1.2 the hook never fires, and there is no test that proves it does.
 6. Lint (after 1.8, 1.9): Run wiki_lint and check _lint_inventory reports the test pages correctly (no false orphans, no false broken links).
 7. Concurrency stress (after 2.5): Two parallel wiki_commit invocations against the same wiki — second should block until first completes; final git log
 should be linear.

 If steps 4 and 6 both produce sane reports, the gate is effective. Without those reports the plugin is loadable but vacuous — the symptom that motivated
 this plan.
