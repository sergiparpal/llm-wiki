Review: llm_wiki memory provider plugin

  Severity ordered. Findings hinge on actual Hermes Agent source at /tmp/hermes-agent (cloned main).

  ---
  BLOCKER

  B1. Tool schemas are double-wrapped — every wiki_* tool is unroutable

  File: __init__.py:271-283 (_fn helper), __init__.py:286-376 (_TOOL_SCHEMAS).

  Plugin returns OpenAI-wrapped schemas:
  {"type": "function", "function": {"name": "wiki_read", ...}}
  But Hermes expects flat schemas ("name" at top level) and wraps them itself. Confirmed three places:
  - agent/memory_manager.py:234-237 — schema.get("name", "") reads from top level. Plugin returns "" here, so _tool_to_provider never indexes any tool name.
  - agent/memory_manager.py:341 — same pattern in get_all_tool_schemas.
  - run_agent.py:1706-1713 — same pattern; also wraps again as {"type": "function", "function": _schema}, producing nested function.function.name for the
  model.

  Net effect: MemoryManager.has_tool("wiki_ingest") is always False, so run_agent.py:8699 never dispatches to the plugin. Bundled honcho
  (plugins/memory/honcho/__init__.py:36-61), mem0, and holographic all use the flat shape — that is ground truth.

  Fix: drop the wrapper. Return entries shaped like:
  {
      "name": "wiki_read",
      "description": "...",
      "parameters": {"type": "object", "properties": {...}, "required": [...]},
  }

  B2. on_compress is dead — the actual hook is on_pre_compress

  File: __init__.py:221-232.

  The plugin overrides on_compress(messages, compression_count). The ABC at agent/memory_provider.py:163 declares on_pre_compress(self, messages: List[Dict])
  -> str, and run_agent.py:8469 and agent/memory_manager.py:415 only ever invoke on_pre_compress. on_compress is never called. The compression-flush feature
  (the entire reason flush_compression_to_raw exists) silently never runs.

  Fix: rename to on_pre_compress(self, messages), drop compression_count (count it internally if needed), and return "" to satisfy the documented -> str
  contract while keeping the side effect.

  B3. prefetch signature crashes every turn

  File: __init__.py:201-205.

  Plugin signature: def prefetch(self, query: str = "") -> str.
  ABC + caller: agent/memory_manager.py:297 calls provider.prefetch(query, session_id=session_id) — session_id is keyword-only and ALWAYS passed. The plugin
  raises TypeError: prefetch() got an unexpected keyword argument 'session_id' on the first turn. prefetch_all swallows the exception (line 300-304) at debug
  level, so the agent keeps running but the wiki context is silently never injected.

  Fix: match the ABC: def prefetch(self, query: str, *, session_id: str = "") -> str.

  B4. handle_tool_call signature missing **kwargs

  File: __init__.py:170.

  ABC: handle_tool_call(self, tool_name, args, **kwargs) -> str.
  Manager: agent/memory_manager.py:371 calls provider.handle_tool_call(tool_name, args, **kwargs). Currently run_agent.py:8700 doesn't pass any extra kwargs,
  so the call accidentally works — but the moment any caller adds context (the codebase already does this for on_memory_write, and is likely to extend it
  here), the plugin breaks. Honcho and mem0 both accept **kwargs.

  Fix: def handle_tool_call(self, tool_name: str, args: dict, **kwargs) -> str.

  B5. README install path is wrong

  File: README.md:36-39.

  Doc tells the user:
  mkdir -p ~/.hermes/plugins/memory
  cp -r llm_wiki ~/.hermes/plugins/memory/llm_wiki
  But plugins/memory/__init__.py:41-47 (memory provider discovery) scans $HERMES_HOME/plugins/<name>/, NOT $HERMES_HOME/plugins/memory/<name>/. The
  _iter_provider_dirs function explicitly enumerates children of plugins/, not plugins/memory/, and _is_memory_provider_dir is the test for whether a child is
   a memory plugin. Following the README's instructions yields a plugin Hermes never sees.

  Fix: install path is ~/.hermes/plugins/llm_wiki/ (no memory/ segment).

  B6. Gate path filters use the wrong prefix — check_frontmatter, check_citations, _apply_edits

  File: gate.py:120 (frontmatter), gate.py:206 (citations), gate.py:445 (apply_edits).

  staged_diff(wiki_root) calls git status with wiki_root = store.root (the storage root, not store.wiki). All paths come back with the wiki/ prefix:
  - A new entity page lands as wiki/entities/karpathy.md, not entities/karpathy.md.
  - The special files are wiki/index.md / wiki/log.md / wiki/overview.md, not index.md / log.md / overview.md.

  Consequences:
  - check_frontmatter skips paths matching {"index.md", "log.md", "overview.md"} — they never match, so the special files always trigger "Missing frontmatter"
   block findings on every commit. Every ingest is rejected.
  - check_citations only fires for paths starting entities/ or concepts/. Those literals don't match anything; the citation check is dead.
  - _apply_edits writes to wiki_root / f.page. If the validator returns "entities/foo.md" (matching the bad filter prefix), the write target is wrong by one
  path component.

  Fix: either (a) call staged_diff with store.wiki so paths come back without wiki/, and adjust callers to know the path is wiki-relative, or (b) keep
  staged_diff(store.root) and update every prefix in gate.py to match wiki/.... Option (b) is the smaller diff but option (a) avoids surprising future
  readers.

  B7. check_links builds an existing set with the wrong shape — every wikilink reads as broken

  File: gate.py:147-151.

  existing = {str(p.relative_to(wiki_root).with_suffix(""))
              for p in wiki_root.rglob("*.md")}
  This produces entries like wiki/entities/karpathy. Wikilinks in the schema are written as [[entities/karpathy]] (per SCHEMA.md:39, :104, :184). The lookup
  target in existing compares entities/karpathy to wiki/entities/karpathy — never matches. Every wikilink is block-flagged. Same bug shows up in
  operations.py:_lint_inventory:510 — lint reports every link as broken.

  Fix: strip the wiki/ prefix before adding to existing, or scope the rglob to store.wiki and resolve relative to it. Be aware of cross-prefix targets like
  [[questions/foo]] so the schema scoping stays consistent.

  B8. _gather_cited_sources cannot find the cited source

  File: gate.py:307-322.

  The wiki convention has wiki/sources/<slug>.md mirroring raw/<kind>/<date>-<title>.md. The wiki-side slug doesn't match the raw filename.
  _gather_cited_sources extracts sources/(slug) and then runs raw_root.rglob(f"{slug}.*") — for a wiki/sources/karpathy-llm-wiki-gist.md page, it greps for
  karpathy-llm-wiki-gist.* under raw/, but the raw file is raw/articles/2026-04-27-karpathy-gist.md. No hit. The validator gets (no cited sources found in
  raw/) for grounding checks. The "claim grounding" check — described as the #1 reason the gate exists — runs blind.

  Fix: record the raw-source path in the wiki source page's frontmatter (e.g., raw_path: raw/articles/2026-04-27-karpathy-gist.md), then read it during
  gathering. Or change the ingest contract so wiki source slug equals raw filename stem.

  B9. check_growth misses brand-new pages

  File: gate.py:172.

  touched = sum(1 for c in diff.values() if c["status"] in {"A", "M"})
  A page written by wiki_write is untracked (not git add-ed) at gate time. Git status reports it as ??, not A. Status ?? isn't in {"A", "M"}, so newly created
   pages don't count. The 5–20 budget never trips on the typical ingest path, which is mostly new pages. (A would be reachable only if something else ran git
  add first; nothing does.)

  Fix: include "??" (or simply c["status"] != "D" and c["after"] is not None).

  B10. gate.commit will fail on hosts without global git config

  File: gate.py:344-346.

  subprocess.run(["git", "-C", str(wiki_root), "commit", "-m", message], check=True)
  No -c user.email / -c user.name. On a fresh container, CI runner, or server without a global identity, every commit raises CalledProcessError. Store.commit
  in operations.py:115-117 does pass -c user.email=hermes@local -c user.name=hermes, but the commit() function in gate.py does not. The gate's commit is on
  the success path of every approved ingest — the plugin will appear to work in dev (where git config is set) and fall over in deployment.

  Fix: plumb the same -c user.email/.name flags into gate.commit, or factor a single helper used by both modules.

  ---
  MAJOR

  M1. Path-prefix check has substring-collision bug

  File: operations.py:132-139 (Store.resolve).

  if not str(target).startswith(str(anchor)):
      raise ValueError(...)
  With anchor = /home/u/.hermes/llm_wiki_data/raw and target = /home/u/.hermes/llm_wiki_data/raw_evil/foo.md, startswith returns True even though raw_evil is
  a sibling, not a child. Real-world exploitability is low (the agent would have to construct an evil sibling directory), but the check is just wrong. Use
  target.is_relative_to(anchor) (Python 3.9+) or os.path.commonpath([target, anchor]) == str(anchor).

  M2. wiki_capture commits while an ingest is in flight, bypassing the gate

  File: operations.py:233-250.

  wiki_capture calls store.commit(...) at the end. commit runs git add -A followed by git commit, sweeping in any uncommitted wiki/ changes from a
  half-finished ingest. The gate never sees them — the partial ingest is silently committed under a capture: ... message. Either reject wiki_capture when
  there are uncommitted wiki/ changes, or restrict the git add to the new raw/ file path (git add raw/<path>).

  M3. _apply_edits is silently lossy

  File: gate.py:440-450.

  if not target.exists():
      continue
  A validator returns N fix findings; only the ones whose target file already exists get applied. applied = > 0 is the pass condition, so as long as one fix
  lands, the ingest commits with the rest dropped on the floor. There's no signal in the verdict that some fixes were skipped. Either fail the whole apply
  when any fix can't land, or strip skipped fixes from verdict.findings before returning so the caller can't believe they were applied.

  M4. lint_every_n_ingests=0 does not disable

  File: __init__.py:80.

  self._lint_every_n = int(config.get("lint_every_n_ingests") or 50)
  0 or 50 → 50. The README documents 0 as "disables." It silently maps to default 50. Fix: value = config.get("lint_every_n_ingests"); self._lint_every_n =
  int(50 if value is None or value == "" else value).

  M5. slug.startswith(("ingest-", "")) is always True

  File: operations.py:405.

  n = store.bump_ingest_count() if slug.startswith(("ingest-", "")) and verdict.decision != "REJECT" else None
  startswith with an empty string in the prefix tuple matches every string. The intent was to count only ingest commits, but the counter advances for
  lint-..., synthesis-..., etc. The Ingest count: N suffix then prints on every non-rejected commit, and the lint_every_n_ingests cadence fires on non-ingest
  counts. Fix: drop the "" from the tuple.

  M6. staged_diff splits NUL-separated rename records incorrectly

  File: gate.py:78-92.

  git status --porcelain=v1 -z emits renamed entries as R  <new>\0<old>\0 — two NUL-separated tokens for one logical rename. The current parser treats each
  NUL chunk as a separate entry, so the <old> half becomes a phantom entry with garbage status (the leading two chars of the path) and the path off by one.
  Agents shouldn't rename mid-ingest, but corrupted parses on a single rename will produce confusing findings. Either filter out R entries explicitly or use
  --porcelain=v2 -z and parse properly.

  M7. Concurrency: single-writer assumption is undocumented in the active path

  File: __init__.py lifecycle, operations.py:Store.commit.

  Two concurrent Hermes sessions on the same profile both reach wiki_commit. There's no advisory lock. Git will reject one of them (fatal: Unable to create
  '.../index.lock'), but the failure surfaces as a generic git error string up to the model. The README mentions this in a one-liner under Limitations; the
  error path doesn't tell the user that's what happened. Either take an fcntl.flock on .git/index for the duration of wiki_commit, or detect the lock-file
  error and return a structured "another session is committing — retry" message.

  M8. prefetch swallows all exceptions silently

  File: __init__.py:201-205.

  def prefetch(self, query: str = "") -> str:
      try:
          return operations.build_prefetch(self._store, query=query)
      except Exception:
          return ""
  With no logging, a misconfigured store, missing wiki/index.md, or unicode error in entities/hermes-agent.md produces silent emptiness. Pair this with B3
  (signature mismatch) and the result is an agent running with NO wiki context and no diagnostic. At minimum log at warning so the operator sees something.

  M9. Validator-error block finding is irrecoverable

  File: __init__.py:115-127, gate.py:298-303.

  If the validator returns malformed JSON or the LLM call raises, the gate emits a validator-error block finding and wiki_commit reverts the diff. Wiki work
  was just thrown away because the checker glitched. A retry (one re-prompt asking the validator to repair the JSON) would be cheap insurance. Or downgrade to   APPROVE_WITH_EDITS warning rather than block so the work isn't lost on transient failures.

  ---
  MINOR

  m1. enrich_turn and on_turn_complete are dead code

  File: __init__.py:208-219.

  Neither method exists in MemoryProvider. Hermes never calls them. The user's documented uncertainty (#5) is resolved: prefetch is the live hook, enrich_turn   is not. sync_turn (not on_turn_complete) is the correct turn-end hook. Delete both.

  m2. register() returns the class instead of using the ctx pattern

  File: __init__.py:384-386.

  The bundled providers do:
  def register(ctx) -> None:
      ctx.register_memory_provider(WikiMemoryProvider())
  The plugin does:
  def register():
      return WikiMemoryProvider
  This happens to load only because plugins/memory/__init__.py:_load_provider_from_dir falls through to a class-walking fallback when register(ctx) produces
  no provider via the collector. That fallback exists for backwards-compat with raw class-only modules; new plugins are expected to use the ctx pattern. The
  fallback also instantiates WikiMemoryProvider() with no constructor — fine here, but fragile if you ever add __init__ args.

  m3. plugin.yaml uses non-canonical keys

  File: plugin.yaml.

  The plugin's manifest declares type: memory_provider, entry_point, and an in-yaml config: block. The general manifest parser at
  hermes_cli/plugins.py:805-817 reads name, version, description, author, requires_env, provides_tools, provides_hooks, kind. Memory plugins are skipped at
  the top level (line 575), so their manifest is parsed by discover_memory_providers at plugins/memory/__init__.py:122-156 — which only reads description.
  None of type, entry_point, config, dependencies are read. The honcho/mem0/holographic manifests use pip_dependencies: and hooks:. Either align with the
  canonical shape or comment that yours is documentation-only.

  m4. _quick_frontmatter_check and _lint_inventory skip-special-file logic uses substring match

  File: operations.py:492 and :519.

  if any(s in rel for s in ("/index.md", "/log.md", "/overview.md")): matches inside arbitrary paths. A page named wiki/entities/log.md.archive.md would
  match. Cosmetic, but use a set of exact relative paths ({"wiki/index.md", "wiki/log.md", "wiki/overview.md"}) instead.

  m5. Frontmatter regex requires literal \n---\n — Windows CRLF will fail

  File: operations.py:43, gate.py:109.

  FRONTMATTER = re.compile(r"\A---\n(.*?)\n---\n", re.DOTALL)
  A file written on Windows with \r\n line endings will not match. The plugin's own wiki_write uses target.write_text(content, encoding="utf-8") which
  respects platform line endings on text mode — but on POSIX the LLM sometimes emits \r\n strings verbatim. Use \r?\n or normalize first.

  m6. _err and tool_error are inconsistent

  File: operations.py:486 vs honcho's use of tools.registry.tool_error.

  The plugin has its own _err returning '{"error": "..."}'. Bundled providers all import tool_error from tools.registry. The behavior is similar but
  tool_error carries optional extras (success=False, etc.) and is what the agent's tool-result parser is tuned for. Use the canonical helper.

  m7. _tail_log parses log dates without timezone awareness

  File: operations.py:545-561.

  datetime.fromisoformat(m.group(1)).timestamp() produces a naive datetime, then .timestamp() interprets it as local time. The cutoff is computed in UTC. On
  non-UTC hosts the 7-day window slides by the offset. Use datetime.combine(date, time.min, tzinfo=timezone.utc) (or skip .timestamp() and compare dates
  directly).

  m8. Tool result type is inconsistent (string vs JSON)

  File: operations.py (wiki_read, wiki_write, wiki_log_append, wiki_capture, wiki_ingest, wiki_query, wiki_lint, wiki_reflect, wiki_commit, wiki_abort).

  The ABC docstring says "Must return a JSON string." The plugin returns plain strings ("wrote {rel}", "log entry appended", "no changes to commit",
  multi-line dedent blobs) AND JSON errors via _err. Honcho/mem0/holographic always return JSON: json.dumps({"result": "..."}). Mixing breaks any downstream
  JSON-parsing of tool results. Wrap every successful return in json.dumps({"result": ...}).

  m9. Frontmatter key set is parsed with a weak regex

  File: operations.py:497, gate.py:126.

  re.findall(r"^(\w+):", m.group(1), re.MULTILINE) extracts the keys but ignores YAML structure. Indented sub-keys (e.g., confidence: nested under another
  field) would be picked up. Multi-line YAML values (folded scalar, list of dicts) are silently ignored. The schema is small enough that this works in
  practice, but if anyone ever writes valid YAML with a nested key, the gate will be confused. Consider importing yaml.safe_load (it's already a soft dep
  elsewhere — the manifest parser uses it).

  m10. _apply_edits logging is silent on partial application

  File: gate.py:440-450.

  When some edits land and others don't (because the target file vanished or suggested_edit is None), nothing is logged. This compounds with M3.

  ---
  NIT

  n1. Hardcoded "wiki" and "raw" literals scattered across modules

  Multiple call sites use must_be_under="wiki" / "raw" as strings. Centralize in constants in operations.py (you already have WIKI_SUBDIRS, add WIKI_DIR =
  "wiki", RAW_DIR = "raw").

  n2. Verdict.to_dict doesn't include the slug

  The verdict JSON returned to the agent omits the slug it just judged. Adding "slug" to to_dict would make rejection records easier to follow.

  n3. _seed_* functions are fine as code but a templates/ dir would scale better

  For >3 seeds you'd want files. Keep as-is for now.

  n4. gate.py:main() and _stub_llm are unused dead paths in the plugin context

  The CLI entry point makes sense if gate.py is run standalone, but the docstring should note the plugin path doesn't go through main.

  n5. Comments occasionally explain WHAT (e.g., operations.py:114-120 "ingest counter")

  Per the project's terse-comment preference, drop comments that restate the section name above them.

  n6. WikiMemoryProvider doesn't override system_prompt_block

  Returning a one-liner like "Wiki memory active. Use wiki_ingest/wiki_query/wiki_lint/wiki_reflect; SCHEMA.md governs." would help the agent discover the
  toolset without leaning entirely on prefetch. The schema content itself is too long to inline, but a pointer + key tools would help.

  n7. flush_compression_to_raw only captures user-side messages

  Assistant context is also useful evidence; you may want to capture both unless you've deliberately decided to avoid auto-citing the agent's own statements
  as source.
